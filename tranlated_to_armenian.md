## 1. Գործադիր ամփոփագիր

PipeScope-ը Python-ի վրա հիմնված հրամանի տողերի անվտանգության թվարկման գործիք է, որը նախատեսված է CI/CD խողովակաշարերի և դրանց օժանդակ ենթակառուցվածքների սխալ կազմաձևումները, հավատարմագրերի բացահայտումը և հայտնի խոցելիությունները հայտնաբերելու համար: Գործիքը միավորում է վերլուծությունը բազմաթիվ մակերեսների վրա՝ տեղական պահեստային ֆայլեր, հեռավոր CI/CD ծառայության վերջնակետեր, կոնտեյներների սահմանումներ, նվագախմբային մանիֆեստներ և վեբ-բացահայտված ուղիներ. (NVD) և հարստացված AI-ի ստեղծած վերականգնման ուղեցույցով:

Համակարգը ինտեգրվում է GitHub Actions-ին որպես բազմակի օգտագործման կոմպոզիտային գործողություն՝ գրելով հարուստ Markdown հաշվետվությունները անմիջապես GitHub Actions-ի ամփոփման վահանակին: Այն աջակցում է արտահանմանը կառուցվածքային JSON և տպագրվող PDF՝ հնարավորություն տալով սպառումը ինչպես ավտոմատացված խողովակաշարերի, այնպես էլ մարդկային վերանայողների կողմից:

PipeScope-ն անդրադառնում է DevSecOps-ի գործիքների լանդշաֆտի լայնորեն ճանաչված բացին. մինչդեռ հաստատված գործիքները, ինչպիսիք են Trivy, Snyk և Semgrep-ը կենտրոնանում են առանձին արտեֆակտների վրա (կոնտեյների պատկերներ կամ սկզբնական կոդը), PipeScope-ն ընդունում է խողովակաշարի վրա կենտրոնացված տեսակետ՝ վերաբերվելով CI/_________________-ին:

## 2. Ներածություն և մոտիվացիա

Ծրագրային ապահովման առաքման արագացումը CI/CD խողովակաշարերով ներմուծել է համակարգային ռիսկի նոր կատեգորիաներ: Խողովակաշարերի սահմանումներ — GitHub Actions աշխատանքային հոսքի ֆայլեր, GitLab CI YAML, Jenkinsfiles — գործարկվում են բարձրացված թույլտվություններով, հաճախ սպառող գաղտնիքներ, ամպային հավատարմագրեր և տեղակայման բանալիներ: Մեկ սխալ կազմաձևումը, ինչպիսին է `pull_request_target`-ի օգտագործումը առանց շրջանակի սահմանափակման, կարող է թույլ տալ անվստահելի պատառաքաղին կարդալ պահեստի գաղտնիքները: Ենթակառուցվածքի շերտում չամրացված Docker բազային պատկերները, բեռնարկղերը, որոնք աշխատում են որպես արմատ, և Kubernetes-ի աշխատանքային ծանրաբեռնվածությունը, որը զուրկ է անվտանգության համատեքստից, ներկայացնում են արտոնությունների մեծացման և կողային շարժման կայուն վեկտորներ:

Չնայած դրան, DevSecOps-ի գործիքների շղթաների մեծ մասը վերաբերվում է խողովակաշարի կոնֆիգուրացիան որպես անվտանգության ավտոմատ վերլուծության շրջանակից դուրս: Անվտանգության ինժեներներին, որոնք իրականացնում են ձեռքով վերանայումներ, բացակայում են CI/CD արտեֆակտներին հարմարեցված մասշտաբային, հետևողական գործիքներ: Արդյունքն այն է, որ կազմակերպությունները հաճախ հայտնաբերում են խողովակաշարերի կարևոր սխալ կազմաձևումներ միայն միջադեպից հետո:

PipeScope-ը ստեղծվել է այս բացը լրացնելու համար. միասնական, միասնական գործիք, որը ինժեները կարող է ուղղորդել պահեստի վրա, կամ կանչել որպես խողովակաշարի մաս, և ստանալ կառուցվածքային, գործող անվտանգության գնահատում ամբողջ CI/CD հարձակման մակերեսի վերաբերյալ:

## 3. Նպատակներ և շրջանակներ

** Առաջնային նպատակները.

1. Հայտնաբերեք անվտանգության սխալ կազմաձևումները CI/CD խողովակաշարի սահմանումներում (GitHub Actions, GitLab CI, Jenkins):
2. Թվարկեք հանրությանը հասանելի վերջնակետերը Jenkins և GitLab սերվերների վրա; հայտնաբերված ծրագրաշարի տարբերակները կապել հայտնի CVE-ների հետ:
3. Բացահայտեք հավատարմագրերի և գաղտնի բացահայտումները աղբյուրի ֆայլերում, խողովակաշարերի սահմանումներում և շրջակա միջավայրի փոփոխականների հայտարարագրերում:
4. Անվտանգության հակաօրինաչափությունների համար վերլուծեք Dockerfile, Docker Compose և Kubernetes մանիֆեստները:
5. Կատարեք վեբ-ուղու հետախուզություն կազմաձևված ծառայության URL-ների նկատմամբ:
6. Տրամադրեք AI-ավելացված, գործող վերականգնողական առաջարկություններ:
7. Թափանցիկ կերպով ինտեգրվել որպես GitHub Գործողությունների քայլ՝ արդյունքները գրելով Գործողությունների ամփոփման վահանակում:
8. Արտահանել մեքենայաընթեռնելի JSON և մարդու կողմից ընթեռնելի PDF:

** Շրջանակից դուրս (v0.1.0):**

- Գործող ծառայությունների դինամիկ անվտանգության փորձարկում (DAST) պասիվ տարբերակի մատնահետքից դուրս:
- Երկուական կամ կախվածության խոցելիության վերլուծություն (ծածկված առկա SCA գործիքներով):
- Խողովակաշարի կոնֆիգուրացիայի ուղղակի մանիպուլյացիա; գործիքը միայն կարդալու է:

PipeScope-ի տարբերակված արժեքը կայանում է ** շրջանակի լայնության մեջ** (ֆայլեր + հեռավոր վերջնակետեր + գաղտնիքներ + CVEs + AI հարստացում + CI ինտեգրում), **զրոյական կախվածության միջուկ** (կանոնների վրա հիմնված AI **it Գործողությունների ստեղնաշարը գործում է առանց __PH_3-ի): աջակցություն** GitHub Step Summary ելքով:

### Տվյալների հոսք

1. Օգտագործողը կանչում է `pipescope [TARGET] [OPTIONS]`:
2. `cli.py`-ը լուծում է `TARGET` (URL → `git clone` ժամանակավոր գրացուցակ, տեղական ուղի → ուղղակի օգտագործում, `None` → միայն հեռահար ռեժիմ) `utils/target.py`-ի միջոցով:
3. Սկանավորող նվագախումբը (`scan.py`) ստանում է `Optional[Path]` արմատ և հեռավոր URL պարամետրեր:
4. Երբ տեղական `root` հասանելի է, բոլոր տեղական սկաներներն աշխատում են հաջորդաբար. նրանց `List[Finding]` արդյունքները միաձուլվում են `ScanResult`-ի:
5. Հեռակառավարվող սկաներներն աշխատում են ինքնուրույն յուրաքանչյուր կազմաձևված URL-ի համար; արդյունքները կցվում են։
6. Բոլոր գտածոները անցնում են `cve_lookup.py`-ով (եթե արդեն հարստացված չեն) և `ai_advisor.py`-ով:
7. Ձևաչափված ելքը ուղարկվում է տերմինալ (Rich), JSON ֆայլ, PDF և/կամ `$GITHUB_STEP_SUMMARY`:

## 6. Մոդուլի մակարդակի ձևավորում և իրականացում

### 6.1 CLI Շերտ — `cli.py`

**Ֆայլ՝ ** `pipescope/cli.py` 
**Պատասխանատվություն.** Մուտքի կետ, փաստարկի սահմանում, ելքի առաքում:

PipeScope-ն օգտագործում է [Typer](https://typer.tiangolo.com/)՝ իր հրամանի տողի ինտերֆեյսը կառուցելու համար՝ հնարավորություն տալով տիպի ծանոթագրված փաստարկների սահմանումներ և `--help` ավտոմատ գեներացում: `TARGET` արգումենտը հայտարարված է որպես `Optional[str]`՝ `None` լռելյայնով, որը թույլ է տալիս գործիքին կանչել առանց տեղական թիրախի, երբ միայն հեռահար սկանավորում է պահանջվում.

```python
def main(
    target: Optional[str] = typer.Argument(None, help="..."),
    jenkins_url: Optional[str] = typer.Option(None, "--jenkins-url"),
    gitlab_url: Optional[str] = typer.Option(None, "--gitlab-url"),
    web_url: Optional[str] = typer.Option(None, "--web-url"),
    ai: bool = typer.Option(True, "--ai/--no-ai"),
    openai_key: Optional[str] = typer.Option(None, "--openai-key"),
    fail_on_high: bool = typer.Option(False, "--fail-on-high/--no-fail-on-high"),
    ...
)
```

Պաշտպանական դրույթը ապահովում է առնվազն մեկ թիրախի տրամադրում նախքան սկանավորումը շարունակելը: Երբ սահմանվում է `$GITHUB_OUTPUT`-ը, գտածոների հաշվարկներն արտահանվում են որպես GitHub Actions ելքային փոփոխականներ (`findings_count`, `high_count`, `critical_count`):

** Դիզայնի հիմնական որոշումները.

- **Միայն հեռակառավարման ռեժիմ.** Երբ `target`-ը `None` է, նվագախումբը բաց է թողնում բոլոր տեղական սկաներները՝ կանխելով `Path`-ի հետ կապված գործարկման ժամանակի սխալները և միացնելով Jenkins կամ GitLab URL-միայն խողովակաշարի կանչերը:
- **Խափանման դրոշը.** `--fail-on-high`-ը հանգեցնում է գործընթացի դուրս գալուն 1 կոդով, եթե առկա են բարձր կամ կրիտիկական բացահայտումներ, ինչը հնարավորություն է տալիս խողովակաշարի դարպասի կիրառումը:
- **AI փոխարկել՝ ** `--ai/--no-ai` վերահսկում է հարստացումը; երբ անջատված է, գտածոները վերադարձվում են այնպես, ինչպես կա՝ նվազեցնելով արտաքին API կախվածությունը օդով բացված միջավայրերից:

---

### 6.2 Սկան նվագարկիչ — `scan.py`

**Ֆայլ՝ ** `pipescope/core/scan.py` 
**Պատասխանատվություն.** Մոդուլի առաքում, արդյունքների համախմբում:

`scan.py`-ը կենտրոնական դիսպետչերն է: Այն ընդունում է `root: Optional[Path]` URL կամընտիր պարամետրերի հետ միասին և հերթով կանչում է յուրաքանչյուր սկաների: Երբ `root is None`, նվագախումբը ստեղծում է `ScanResult` `"mode": "remote-only"` մետատվյալներով և անմիջապես անցնում է հեռավոր մասնաճյուղի սկաներներին:

```python
if root is not None:
    res = run_cicd_enumeration(root, ...)
    # ... all local scanners ...
else:
    res = ScanResult(target="(no local target)", findings=[])
    res.metadata["mode"] = "remote-only"
```

Կենտրոնացնելով առաքումն այստեղ՝ առանձին սկաներները մնում են քաղաքացիություն չունեցող գործառույթներ, որոնք ընդունում են ուղին և վերադարձնում `List[Finding]`՝ հավատարիմ մնալով մեկ պատասխանատվության սկզբունքին:

---

### 6.3 CI/CD դետեկտոր — `cicd_detector.py`

**Ֆայլ՝ ** `pipescope/core/cicd_detector.py` 
**Պատասխանատվություն.** Հայտնաբերել CI/CD կազմաձևման ֆայլերը պահեստում; գնահատել TOML կանոնների փաթեթները արդյունահանված դիտարկումների համեմատ. կանչել գաղտնի հայտնաբերում.

`detect_cicd_files()`-ը կատարում է դետերմինիստական ​​անցում.

- `<root>/.github/workflows/*.y*ml` → `GitHubActions`
- `<root>/.gitlab-ci.yml` → `GitLabCI`
- `<root>/Jenkinsfile` → `Jenkins`

Յուրաքանչյուր հայտնաբերված ֆայլ այնուհետև ուղարկվում է իր համապատասխան դիտարկման արդյունահանողին և կանոնների փաթեթի գնահատողին.

```
GitHubActions → gha_observe.extract_gha_observations()
                + eval_rules(observations, GHA_RULES)
GitLabCI      → gitlab_observe.extract_gitlab_observations()
                + eval_rules(observations, GITLAB_RULES)
Jenkins       → jenkins_observe.extract_jenkins_observations()
                + eval_rules(observations, JENKINS_RULES)
```

Բացի այդ, `cicd_detector.py`-ն ուղղակիորեն կանչում է՝
- `scan_for_secrets_detect_secrets()` — հայտնաբերել-գաղտնիացնում է պահեստի արմատի գրադարանի սկանավորումը:
- `scan_for_weak_secrets()` - հանձնարարական-օրինաչափության սկաներ՝ չնչին գուշակելի արժեքների համար:

CI/CD-հատուկ գաղտնի սկանավորման այս համատեղ տեղակայումը դետեկտորի ներսում ապահովում է, որ համատեքստի նկատմամբ զգայուն բոլոր ստուգումները (օրինակ՝ գաղտնիքները աշխատանքային հոսքի ֆայլերում) գնահատվում են մեկ անցումով:

### 6.4 Կանոնների շարժիչ — `rules_eval.py` և `rule_engine.py`

**Ֆայլեր՝ ** `pipescope/core/rules_eval.py`, `pipescope/core/rule_engine.py`

Այս երկու մոդուլներն իրականացնում են կանոնների գնահատման լրացուցիչ պարադիգմներ:

#### `rules_eval.py` — Դիտարկման վրա հիմնված TOML կանոնների գնահատող

Գնահատում է սկալյար դիտարկումների բառարանը (քաղված CI/CD կազմաձևման ֆայլից) TOML կանոնների ցանկի համեմատ: Աջակցում է չորս օպերատորների.

| Օպերատոր | Իմաստաբանություն |
|----------|-----------|
| `contains` | `value ∈ observations[field]` (ցուցակային անդամակցություն) |
| `equals_ci` | մեծատառերի անզգայուն տողերի հավասարություն |
| `regex` | `re.search(value, observations[field])` |
| `any_true` | ճիշտ է, եթե `observations[field]`-ի որևէ տարր ճշմարտացի է |

Այս դիզայնը անջատում է դիտարկման արդյունահանման տրամաբանությունը գնահատման տրամաբանությունից՝ թույլ տալով նոր կանոնների փաթեթներ ավելացնել՝ առանց Python կոդը փոփոխելու։

#### `rule_engine.py` - Ֆայլի բովանդակության ռեգեքս սկաներ

Բեռնում է `pipescope/rules/secrets.toml`-ը և համընկնում է ռեգեքսի օրինաչափությունների հետ ֆայլի բովանդակության ֆիքսված ընդլայնումների համար (`.java`, `.js`, `.ts`, `.py`, `.yml`, `.yaml`, `.env`): Յուրաքանչյուր համընկնումը տալիս է եզրակացություն CWE հղումով և գծային ապացույցով՝ TOML կանոնի սահմանումից:

### 6.5 Դիտորդական արդյունահանողներ

**Ֆայլեր՝ ** `pipescope/core/gha_observe.py`, `pipescope/core/gitlab_observe.py`, `pipescope/core/jenkins_observe.py`

Յուրաքանչյուր արդյունահանող վերլուծում է CI/CD կոնֆիգուրացիայի ֆայլը և արտադրում է հարթ `Dict[str, Any]` նորմալացված դիտարկումներ, որոնք հարմար են `rules_eval.eval_rules()`-ի կողմից գնահատման համար:

#### GitHub գործողություններ (`gha_observe.py`)

Արտադրված դիտարկումներ.

#### Շերտ 1 — `gitleaks_adapter.py`

Փաթաթում է [gitleaks](https://github.com/gitleaks/gitleaks) գործիքը ենթապրոցեսի միջոցով՝ կանչելով այն `--no-git --report-format json`-ով՝ սկանավորելու աշխատանքային գրացուցակը, այլ ոչ թե git պատմությունը: Ելքի կոդերը `0` (առանց գտածոների) և `1` (առկա են գտածոներ) երկուսն էլ համարվում են վավեր. ցանկացած այլ ելքի ծածկագիր համարվում է գործիքի ձախողում և նրբագեղորեն ճնշվում է `is_gitleaks_available()` պահակախմբի միջոցով:

Gitleaks-ի գտածոները քարտեզագրված են `Finding` օբյեկտների վրա՝ `PS-REF-GITLEAKS-{ruleID}` ձևի ID-ներով: Խստության հանձնարարություն.
- CRITICAL — կանոնի նկարագրությունը պարունակում է «մասնավոր» կամ «բանալի»
- HIGH - մնացած բոլոր կանոնները

#### Շերտ 2 — `secrets_scanner.py`

Ինտեգրում է [detect-secrets](https://github.com/Yelp/detect-secrets) գրադարանը՝ օգտագործելով իր Python API:

```python
from detect_secrets import SecretsCollection
from detect_secrets.settings import default_settings

with default_settings():
    secrets = SecretsCollection()

```

#### Շերտ 4 — `weak_secrets.py`

Թիրախավորում է ավելի նեղ խնդիր. սովորաբար օգտագործվող թույլ արժեքներ (օրինակ՝ `"changeme"`, `"secret"`, `"1234"`), որոնք վերագրվում են իմաստային առումով համապատասխան ստեղներին (`secret`, `jwt_secret`, `session_secret`, `cookie_secret`): Հայտնաբերումն օգտագործում է կազմված հանձնարարական ռեգեքս.

```python
ASSIGNMENT_RE = re.compile(r"""(?ix)
    \b(?P<key>secret|session_secret|...)\b
    \s*[:=]\s*
    (?P<quote>['"])(?P<value>[^'"\n]{1,64})(?P=quote)
""")
```

Փաստաթղթերի օրինակներից կեղծ դրական արդյունքները նվազեցնելու համար մեկ տողով մեկնաբանությունների տողերը հանվում են `.py`, `.js`, `.ts`, `.yml`, `.yaml`, `.env` ֆայլերից՝ նախքան համապատասխանելը:

**Ֆայլ՝ ** `pipescope/core/web_recon.py`

Վեբ հետախուզության մոդուլը կատարում է բառարանի վրա հիմնված ուղիների թվարկում՝ ըստ կազմաձևված URL բազայի՝ օգտագործելով `concurrent.futures.ThreadPoolExecutor`: Երկու ներկառուցված բառացանկ միավորված են `pipescope/wordlists/`-ում.

- `common.txt` - ընդհանուր նշանակության ուղիներ (ադմինիստրատորի վահանակներ, չափումներ, առողջության ստուգումներ, API փաստաթղթեր)

HTTP պատասխանները `2xx` միջակայքում առաջացնում են `Finding` խստությամբ՝ հիմնված ուղու զգայունության դասակարգման վրա: Թեմաների քանակը, բառացանկի ընտրությունը և URL հիմքը կարգավորելի են CLI դրոշների միջոցով (`--threads`, `--wordlist-name`):

### 6.11 CVE Փնտրում — `cve_lookup.py`

**Ֆայլ՝ ** `pipescope/core/cve_lookup.py`

CVE որոնման մոդուլը հարցում է անում [NIST NVD API v2.0](https://services.nvd.nist.gov/rest/json/cves/2.0)՝ ծրագրային արտադրանքի/տարբերակի զույգերը հայտնի CVE նույնացուցիչների մեջ լուծելու համար: Դիզայնի հիմնական ասպեկտները.

**Caching.** API զանգերը հիշվում են `functools.lru_cache(maxsize=256)`-ի միջոցով ներքին `_search_cves()` ֆունկցիայի մեջ՝ կանխելով ցանցի կրկնվող հարցումները նույն արտադրանքի/տարբերակի համակցության համար մեկ սկանավորման ընթացքում:

**Արտադրանքի նորմալացում.** Քարտեզագրման աղյուսակը նորմալացնում է սովորական ոչ պաշտոնական անունները իրենց NVD CPE արտադրանքի տողերի համար.

```python
_PRODUCT_MAP = {
    "python": "python",
    "nginx": "nginx",
    "jenkins": "jenkins",
    "gitlab": "gitlab",
    "node": "node.js",
    ...
}
```

**Տարբերակի մշուշոտ համընկնումը.** Երբ ճշգրիտ տարբերակի համընկնումը չի տալիս CVE-ներ, մոդուլը կասկադով անցնում է աստիճանաբար ավելի կոպիտ տարբերակի ձևաչափերով (major.minor.patch → major.minor → major)՝ բարելավելու կենդանի ծառայություններից արդյունահանված մասնակի տարբերակի տողերի հետկանչը:

**Պատկերի նորմալացում.** Docker-ի բազային պատկերի հղումները (օրինակ՝ `python:3.11-slim`, `nginx:1.25.3-alpine`) վերլուծվում են `_normalize_image()`-ով` նախքան CVE որոնումը հեռացնելու բաշխման վերջածանցները:

### 6.13 GitHub Գործողությունների ինտեգրում — `github_summary.py`

**Ֆայլ՝ ** `pipescope/core/github_summary.py`

GitHub Summary մոդուլը գրում է Markdown փաստաթուղթ ֆայլի ուղու վրա, որը վկայակոչված է `$GITHUB_STEP_SUMMARY`-ի կողմից, GitHub-ի մեխանիզմը՝ հատուկ բովանդակության վրա Գործողությունների գործարկման ամփոփման վահանակում: Ֆունկցիան անվտանգ է, երբ `$GITHUB_STEP_SUMMARY`-ը կարգավորված չէ (այսինքն՝ GitHub Actions վազորդից դուրս):

Ստեղծված Markdown-ը ներառում է.

1. **Տեղեկատվական վահանակ** — սկանավորեք թիրախը, ժամանակի դրոշմակնիքը, ըստ խստության հաշվումները գտնելու:
2. **AI Ամփոփագիր ** — պարբերություն `build_ai_summary()`-ից:
3. **Խստության բաշխում** — ASCII գծապատկեր՝ էմոջիի խստության ցուցիչներով:
4. **Գտածոների աղյուսակ** — ID, խստություն, վերնագիր, AI առաջարկություն յուրաքանչյուր գտածոյի համար:
5. **CVE հղումներ** — բոլոր CVE ID-ների կրկնօրինակված ցուցակ, որոնք հղում են արվել գտածոներին:
6. **Անցում/խափանման ստորագիր** — կանաչ կամ կարմիր կրծքանշան, որը ցույց է տալիս սկանավորման արդյունքը:

Բացի այդ, `print_ai_panel()`-ը նույն տեղեկատվությունը տրամադրում է տերմինալին [Rich](https://github.com/Textualize/rich) գրադարանի միջոցով՝ օգտագործելով գունավոր կոդավորված `Panel` և `Table` վիջեթները՝ ինտերակտիվ նիստերում ընթեռնելի լինելու համար:

```python
@dataclass
class Finding:

@dataclass
class ScanResult:

    def to_dict(self) -> Dict[str, Any]: ...
```

## 7. TOML Կանոնների փաթեթի համակարգ

PipeScope-ի CI/CD վերլուծությունը պայմանավորված է TOML կանոնների փաթեթներով, որոնք գտնվում են `pipescope/rules/`-ում: Այս դիզայնը առանձնացնում է անվտանգության գիտելիքները (կանոնները) գնահատման տրամաբանությունից (կոդ)՝ թույլ տալով անվտանգության ինժեներին նպաստել կամ թարմացնել կանոնները՝ առանց Python աղբյուրը փոփոխելու:

### Կանոնների փաթեթի ֆայլեր

| Ֆայլ | Դոմեն |
|------|--------|
| `cicd_github_actions.toml` | GitHub Actions աշխատանքային հոսքի անվտանգություն |
| `cicd_gitlab.toml` | GitLab CI/CD կազմաձևման անվտանգություն |
| `cicd_jenkins.toml` | Jenkinsfile անվտանգություն |
| `secrets.toml` | Կոշտ կոդավորված հավատարմագրերի նմուշներ |

### Կանոնների կառուցվածքը

```toml
[[rule]]
id = "PS-GHA-TRIG-001"
severity = "HIGH"
title = "Dangerous trigger: pull_request_target"
field = "triggers"
op = "contains"
value = "pull_request_target"

```

- `id` — եզակի, անվանատարածքի նույնացուցիչ՝ կրկնօրինակման և AI հարստացման որոնման համար:
- `severity` — մեկը `CRITICAL`, `HIGH`, `MEDIUM`, `INFO`-ից:
- `field` - դիտարկման բանալին գնահատելու համար (արտադրված է արդյունահանողի կողմից):

- `value` - օպերանդ օպերատորի համար:
- `recommendation` - մարդու կողմից ընթեռնելի վերականգնողական ուղեցույց:

### Կրիտիկական փաթեթավորման ուղղում

CI վավերացման ընթացքում բարձրացվեց `FileNotFoundError`, քանի որ անիվը կառուցված էր առանց `rules/` և `wordlists/` գրացուցակների: Արմատային պատճառ. `setuptools`-ն ավտոմատ կերպով չի ներառում ոչ Python ֆայլերը, եթե հստակորեն հայտարարված չէ: Բանաձև:

```toml
[tool.setuptools]
include-package-data = true

[tool.setuptools.package-data]
pipescope = [
  "rules/*.toml",
  "wordlists/*.txt",
]
```

Սա հաստատվել է՝ ստուգելով կառուցված անիվի պարունակությունը.

```bash
python -m build
unzip -l dist/pipescope-0.1.0-py3-none-any.whl | grep -E 'rules|wordlists'
```

## 13. Հատկանշական տեխնիկական որոշումներ

### 1. Ընտրովի թիրախային փաստարկ

`TARGET` CLI արգումենտը ընտրովի է (`typer.Argument(None)`)՝ աջակցելու միայն հեռահար սկանավորման ռեժիմներին (օրինակ՝ Ջենկինսի URL սկանավորումը խողովակաշարում՝ առանց կլոնավորված պահեստի): Պահակային տրամաբանությունը ապահովում է փոխադարձ բացառիկության պահպանումը. `TARGET`, `--jenkins-url`, `--gitlab-url` կամ `--web-url`-ից առնվազն մեկը պետք է տրամադրվի:

```python
if target is None and not any([web_url, jenkins_url, gitlab_url]):
    raise typer.BadParameter("Provide a target path/URL or at least one remote URL.")
```

### 2. Բազմաշերտ գաղտնի հայտնաբերում նրբագեղ քայքայմամբ

Գաղտնի հայտնաբերման չորս շերտի ճարտարապետությունը (gitleaks → detect-secrets → TOML regex → թույլ արժեքի նշանակում) ընտրվել է առավելագույնի հասցնելու հայտնաբերման ծածկույթը: Յուրաքանչյուր շերտ նրբորեն քայքայվում է. gitleaks-ի անհասանելի լինելը չի ​​խանգարում հայտնաբերել գաղտնիքների գործարկմանը և հակառակը: Սա կարևոր է GitHub Actions միջավայրերի համար, որտեղ կամայական գործիքների տեղադրումը կարող է սահմանափակվել:

### 3. TOML-Առանձնացված կանոնների փաթեթներ

Անվտանգության կանոնները Python-ի տրամաբանությունից TOML ֆայլերի առանձնացնելը հնարավորություն է տալիս «պլագինանման» ընդարձակման մոդելի: Անվտանգության ինժեներները կարող են նոր ստուգումներ կատարել՝ ավելացնելով TOML գրառումներ՝ չհասկանալով Python-ի ներքին տարրերը: `rules_eval.py` գնահատողի չորս ընդհանուր օպերատորները ծածկում են CI/CD սխալ կազմաձևման ստուգումների ճնշող մեծամասնությունը:

### 4. LRU-Քեշավորված CVE որոնումներ

CVE որոնումները պահվում են `_search_cves()` մակարդակում: Մեծ պահեստի սկանավորումը կարող է վկայակոչել նույն բազային պատկերը բազմաթիվ Dockerfiles-ներում. քեշավորումը կանխում է արագության սահմանափակումը NVD API-ով և նվազեցնում է սկանավորման ընդհանուր տևողությունը մոտավորապես 60–80%-ով ներկայացուցչական փորձարկման դեպքերում:

### 5. Kubescape Go Source Fallback

Փոխանակ ձախողվելու, երբ kubescape-ը տեղադրված չէ, գործիքը վերադառնում է `go run .`-ի գործարկմանը՝ ներկառուցված `kubescape/` Go source գրացուցակի վրա: Սա հնարավորություն է տալիս Kubernetes-ին սկանավորել ցանկացած միջավայրում՝ տեղադրված Go-ով, առանց նախապես կառուցված kubescape երկուականի պահանջելու:

### 6. Փաթեթի տվյալների ընդգրկման ուղղում

GitHub Actions-ում հանդիպած `FileNotFoundError`-ը, երբ TOML կանոնների ֆայլերը բացակայում էին տեղադրված անիվից, հանգեցրեց լուրջ շտկմանը `pyproject.toml`-ում: Սա ընդգծեց անիվի պարունակությունը որպես CI խողովակաշարի մաս վավերացնելու կարևորությունը, այլ ոչ թե հիմնվելով բացառապես տեղական զարգացման փորձարկումների վրա:

### 7. SSH Multi-Account Git-ի կազմաձևում

Մշակման ընթացքում push գործողությունները ձախողվեցին SSH երկու GitHub հաշիվների միջև եղած անհասկանալիության պատճառով: Բանաձևը` `Host` կեղծանունը `~/.ssh/config`-ում, որը ցույց է տալիս բանալու նույնականացման ճիշտ ֆայլը, զուգորդված SSH ձևաչափով հեռակառավարվող URL-ի հետ, ստանդարտ օրինակ է բազմահաշվային git-ի կառավարման համար, որը արժանի է փաստաթղթավորման բազմաթիվ ինքնություններ կառավարող թիմերի համար:

## 15. Եզրակացություն

PipeScope-ը ցույց է տալիս, որ համապարփակ CI/CD անվտանգության վերլուծությունը կարող է իրականացվել որպես թեթև, կախվածության նվազագույն չափով Python գործիք, որը թափանցիկ կերպով ինտեգրվում է նույն խողովակաշարերի մեջ, որը վերլուծում է: CI/CD կազմաձևման ֆայլերը, կոնտեյների սահմանումները, Kubernetes-ի մանիֆեստները, հեռավոր ծառայության վերջնակետերը և սկզբնական կոդի գաղտնիքները որպես միասնական հարձակման մակերես՝ այն ապահովում է անվտանգության տեսանելիություն, որը նեղ շրջանակի գործիքները (պատկերի սկաներներ, SCA գործիքներ, __PH_5_) չեն հասցեագրում:

Ծրագիրն ուներ համակարգերի ինժեներական հմտությունների լայն շրջանակ՝ CLI նախագծում և փաթեթավորում, ենթագործընթացների ինտեգրում արտաքին անվտանգության գործիքների հետ, REST API սպառում քեշավորման միջոցով, YAML/TOML վերլուծություն, միաժամանակյա HTTP սկանավորում, տվյալների ինտեգրում և PH_6, GitHub Actions աշխատանքային հոսքի ճարտարագիտություն: Իրական գործառնական խնդիրների լուծումը՝ SSH բազմահաշվի բանալիների կառավարում, Python անիվի փաթեթավորում ոչ կոդային ակտիվների, ներկառուցված git պահեստի կոնֆլիկտներ, արտացոլում է պրագմատիկ խնդիրների լուծման այն տեսակը, որը բնութագրում է արտադրության մակարդակի գործիքակազմի զարգացումը:

TOML կանոնների փաթեթի ճարտարապետությունը ապահովում է հստակ ընդարձակելիության մոդել ապագա անվտանգության գիտելիքների ներդրման համար, մինչդեռ գաղտնի հայտնաբերման բազմաշերտ ենթահամակարգը և NVD CVE հարաբերակցությունը գործիքին տալիս են իրական հայտնաբերման խորություն: Առաջին կարգի GitHub Actions-ի ինտեգրումը Step Summary ելքի հետ երաշխավորում է, որ բացահայտումները հայտնվում են հենց այնտեղ, որտեղ մշակողները արդեն որոշումներ են կայացնում իրենց կոդի վերաբերյալ:

PipeScope v0.1.0-ը ներկայացնում է ամուր հիմք արտադրության անվտանգության գործիքի համար: Վերը նշված սահմանափակումները և ապագա աշխատանքային կետերը հստակ ճանապարհային քարտեզ են ապահովում ձեռնարկության ընդունման համար անհրաժեշտ ծածկույթի խորության և ինտեգրման լայնության ուղղությամբ:
